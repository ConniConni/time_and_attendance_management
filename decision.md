**決定日**: 2025年12月15日

**承認部門**: [人事部・システム部等を記入]

## 1. 決定の概要

### 1.1 採用方式の基本方針

勤怠管理システムの残業申請方式として、**「事前申請を原則としながら、正当な理由がある場合は事後申請も認める」** バランス型のアプローチを採用する。

### 1.2 主要な特徴

| 項目 | 内容 |
|---|---|
| **基本原則** | 事前申請（前営業日17:00までに申請） |
| **例外措置** | 緊急時は当日・事後申請を許可（退勤後48時間以内） |
| **承認フロー** | 直属上司による承認（事後申請は理由確認を重視） |
| **実績管理** | 申請時間と実績時間の差異を自動照合（30分以上で管理者確認） |
| **抑止策** | 事後申請が月3回以上で管理者に警告通知 |
| **人事部承認** | 月60時間超過時は人事部の二段階承認が必要 |

---

## 2. 採用理由

### 2.1 主要な採用理由

本方式を採用する理由は以下の通りです。

#### 理由1: 一般的な企業特性に適合
- 当社は製造業のような厳格な計画性も、スタートアップのような極端な柔軟性も必要としない、一般的な企業である
- 適度な統制と柔軟性のバランスが、当社の業務特性に最も適している
- 過度に厳格な運用は現場の反発を招き、過度に緩い運用は管理が困難になる

#### 理由2: 実用性と管理のバランス重視
- **実用性**: 緊急時の顧客対応、システム障害対応など、予測不可能な業務に柔軟に対応できる
- **管理性**: 原則として事前申請を求めることで、労働時間の計画性を確保し、36協定違反を防ぐ
- **法令遵守**: 事後申請を認めることで、未払い残業のリスクを回避し、労働基準法に準拠
- **監査対応**: 事前申請原則により、監査時に適切な労働時間管理を証明できる

#### 理由3: 実装難易度・運用負荷が現実的
- **開発期間**: 18週間（約4.5ヶ月）で実現可能
- **開発コスト**: 約900万円（段階的導入により初期600万円に削減可能）
- **運用負荷**: 管理者の承認業務は増加するが、一括承認機能等で軽減可能
- **段階的導入**: フェーズを分けて導入することで、リスクを最小化

### 2.2 他方式を不採用とした理由

#### 案A（厳密型）を不採用とした理由
- ✗ 緊急時の業務対応が困難（顧客対応、障害対応等）
- ✗ 従業員の不満が大きく、サービス残業を誘発するリスク
- ✗ 未申請残業を無効とすることで、未払い残業訴訟のリスクが高い
- ✗ 柔軟性の欠如により、実態と制度が乖離する可能性

#### 案C（柔軟型）を不採用とした理由
- ✗ 事前の労働時間把握ができず、予算管理が困難
- ✗ 長時間労働の歯止めがなく、健康管理上のリスク
- ✗ 36協定違反（月60時間、80時間上限）を事後的にしか把握できない
- ✗ 事後管理の正当性について、監査時の説明が困難
- ✗ 水増し請求など、不正のリスクが高い

## 3. 期待される効果

### 3.1 法令遵守の向上

| 効果 | 詳細 |
|---|---|
| **未払い残業の防止** | 事後申請により、全ての残業が記録・申請され、賃金が適正に支払われる |
| **36協定の遵守** | 事前申請原則により、月60時間超過前に人事部承認を得る仕組み |
| **労働時間の正確な記録** | 実績と申請の突合により、正確な労働時間が記録される |

### 3.2 業務効率の向上

| 効果 | 詳細 |
|---|---|
| **緊急対応の柔軟性** | 顧客対応、障害対応など予測困難な業務に柔軟に対応可能 |
| **計画性の維持** | 事前申請原則により、労働時間の計画性が確保される |
| **承認業務の効率化** | 一括承認機能により、管理者の負担を軽減 |

### 3.3 労務管理の改善

| 効果 | 詳細 |
|---|---|
| **長時間労働の抑止** | 事後申請頻度の監視により、恒常的な残業を早期発見 |
| **健康管理** | 事前に残業時間を把握し、過度な労働を防止 |
| **透明性の向上** | 申請・承認の履歴が明確に記録され、説明責任を果たせる |

### 3.4 従業員満足度の向上

| 効果 | 詳細 |
|---|---|
| **柔軟性の確保** | 緊急時に柔軟に対応できるため、従業員の不満が少ない |
| **公平性** | 正当な理由があれば事後申請が認められ、公平感がある |
| **サービス残業の撲滅** | 全ての残業が申請できるため、サービス残業がなくなる |

## 5. 運用ルール

### 5.1 事前申請の原則

**適用ケース**: 通常の残業（予測可能な業務）

**申請期限**: 残業実施日の前営業日17:00まで

**申請内容**:
- 残業予定日
- 予定開始時刻・終了時刻
- 残業理由（業務内容）

**承認期限**: 申請から5営業日以内

---

### 5.2 事後申請の例外措置

**適用ケース（緊急事由の例）**:
- 顧客からの緊急依頼（クレーム対応、仕様変更等）
- システム障害・トラブル対応
- 突発的な業務発生（上司からの緊急指示等）
- 会議の延長による遅延

**申請期限**: 退勤打刻後48時間以内

**申請内容**:
- 残業実施日
- 実際の開始時刻・終了時刻
- 緊急理由（詳細に記載、100文字以上推奨）

**承認期限**: 申請から5営業日以内

**注意事項**:
- 月3回以上の事後申請は管理者に警告通知
- 恒常的な事後申請は業務改善の対象

---

### 5.3 申請と実績の差異管理

**差異の許容範囲**:
- 30分未満: 自動承認（システムで吸収）
- 30分以上: 管理者による確認・承認が必要

**超過時の対応**:
- 実績が申請を大幅に超過（1時間以上）: 追加申請を促す
- 実績が申請より短い: 承認時間を実績に合わせて調整

---

### 5.4 月60時間超過時の対応

**人事部承認フロー**:
1. 月次残業時間が累計60時間に到達
2. 次の残業申請時にシステムが自動検知
3. 直属上司が一次承認
4. 人事部が二次承認（36協定特別条項の確認）
5. 両方の承認が揃って確定

**人事部の確認事項**:
- 36協定の特別条項が適用可能か
- 月100時間、年720時間の上限に抵触しないか
- 健康管理上の問題はないか
- 業務改善の余地はないか

## 6. 運用開始後の管理

### 6.1 KPI（重要業績評価指標）

| KPI | 目標値 | 測定頻度 |
|---|---|---|
| **事後申請率** | 10%以下 | 月次 |
| **未申請残業の発生件数** | 0件 | 月次 |
| **承認期限超過率** | 5%以下 | 月次 |
| **月60時間超過者数** | 全従業員の5%以下 | 月次 |
| **システム稼働率** | 99.5%以上 | 月次 |
| **従業員満足度（勤怠管理）** | 3.5点以上（5点満点） | 四半期 |

### 6.2 定期レビュー

**月次レビュー**:
- 事後申請の頻度・理由の分析
- 未申請残業の発生状況
- KPIの達成状況

**四半期レビュー**:
- 運用ルールの妥当性検証
- 業務改善の必要性判断
- システム改善要望の収集

**年次レビュー**:
- 方式全体の評価・見直し
- 法令改正への対応確認
- 次年度の改善計画策定

### 6.3 改善サイクル

```
【PDCAサイクル】
Plan（計画）
  ↓
Do（実行）: 運用開始、データ収集
  ↓
Check（評価）: 月次・四半期レビュー
  ↓
Act（改善）: ルール改定、システム改修
  ↓
（繰り返し）
```

- 本決定書（decision.md）

- 運用ルール・ガイドライン

- 事後申請の常態化リスク

**対応策**:
- 事後申請率のデータを提示（目標10%以下）
- 恒常的な事後申請者への改善指導の実績
- 36協定の遵守状況（月60時間超過時の人事部承認）
